## 1.1 ecommerce-agentic-rag-master — 轻量级 Agentic RAG

**来源**：独立开发者项目 | **语言**：Python | **框架**：LangChain + ChromaDB

### 项目概述

这是一个相对简单的电商 RAG Agent 项目，核心实现约 300 行代码，展示了如何用 LangChain 构建带工具调用能力的 Agent。特点是**纯命令行界面**，适合作为理解 Agentic RAG 工作原理的教学级项目。

### 技术架构

```
用户查询 → LangChain Agent → ProductSearchTool (工具)
                              │
                              ↓
                    ChromaDB 向量检索 + 元数据过滤
                              │
                              ↓
                    GPT-4 生成自然语言回答
```

```python
workflow.add_node("tool_node", ToolNode(tools=tools))  # 工具执行节点
```

#### 2. 工具节点异步执行（亮点）

```python
class ToolNode:
    async def __call__(self, state, config):
        state["logs"] = state.get("logs", [])
        state["processing_status"] = "processing"

        # 发射中间状态到前端
        await copilotkit_emit_state(config, state)

        # 执行工具
        for tool_call in tool_calls:
            result = await tool.ainvoke(input=tool_args["query"])
            # 添加工具结果到消息
            tool_message = ToolMessage(
                content=str(result),
                tool_call_id=tool_call_id
            )
            state["messages"].append(tool_message)

            # 再次发射状态
            await copilotkit_emit_state(config, state)

        return state
```

**优势**：异步执行 + 状态实时推送，支持前端展示 Agent 思考过程。

```python
class LLMChatbot:
    @staticmethod
    async def get_response(user_id: str, user_cep: str, user_message: str, ...):
        # 第一次调用：让 GPT 决定是否调用工具
        completion_response = await LLMHandler.call_completions_api(messages, tools)

        tool_calls = completion_response.get("tool_calls", None)

        if tool_calls is not None:
            # 处理工具调用
            tools_output_messages = await LLMChatbot._process_tool_calls(
                user_id, user_cep, tool_calls
            )

            # 第二次调用：不带工具，让 GPT 生成最终回答
            messages = [{"role": "system", ...}] + history + [tool_call_message] + tools_output_messages
            completion_response = await LLMHandler.call_completions_api(messages)
```

**优势**：清晰的两次调用模式，第一次让 GPT 决策，第二次生成回答。

### ✅ 优势总结

| 优势                         | 说明                                        |
| -------------------------- | ----------------------------------------- |
| **购物车完整实现**                | add/remove/update/summary 全部实现，含业务规则      |
| **GPT-4 Function Calling** | 不依赖 LangChain，直接用 Function Calling 实现工具调用 |
| **Rasa NLU**               | 强大的意图识别和实体抽取，处理自然语言变体能力强                  |

| **多服务架构**                  | 模块解耦，便于独立扩展                               |

### ❌ 缺陷总结（针对课题）

| 缺陷                        | 说明                        | 对课题的影响             |
| ------------------------- | ------------------------- | ------------------ |
| **Rasa 学习成本高**            | 需编写 YAML 配置文件，理解 Rasa 概念  | 🔴 陡峭的学习曲线         |
| **Streamlit 前端**          | 不是原生 Android 应用           | 🔴 需自建 Android 客户端 |
| **GPT-4 强依赖**             | 需 OpenAI API 密钥           | 🟡 可替换为豆包          |
| **无流式输出**                 | 一次性返回完整回答                 | 🔴 需添加 SSE         |
| **便利店场景**                 | 容积限制等业务逻辑不适合通用电商          | 🟡 需调整业务规则         |
| **多服务部署复杂**               | 4 个服务需要 Docker Compose 编排 | 🟡 部署成本高           |
| **无多模态**                  | 仅文本对话，无图片/语音              | 🔴 需自建             |
| **Function Calling 两次调用** | 延迟较高，每次对话至少 2 次 LLM 调用    | 🟡 成本翻倍            |

### 🎯 对课题的借鉴价值

**高**。购物车实现和 Function Calling 模式是最佳参考。

**建议借鉴的部分**：

1. **CartHandler 完整实现**：可直接参考增删改查逻辑和业务规则
2. **Function Calling 两次调用模式**：GPT 决策 → 执行 → GPT 回答
3. **Guardrails 实现**：违禁词过滤、品牌白名单
4. **Redis 会话存储**：跨会话数据持久化
5. **工具调用排序**：处理操作依赖关系

**不建议直接使用的原因**：

- Rasa 框架学习成本高
- 多服务架构对课题复杂度偏高
- Streamlit 前端无法用于 Android

## 三、结论

三个新增项目中，**ecommerce-rag-agent-main** 最适合作为课题的核心架构参考，其 LangGraph 工作流模式、状态管理、向量检索逻辑都可直接复用。**retailGPT-main** 的购物车实现是最大亮点，可作为业务层参考。**ecommerce-agentic-rag** 适合作为理解 Agentic RAG 原理的入门项目。

建议按照以下顺序学习：

1. **先看 ecommerce-agentic-rag**：理解 Agent + Tool 的基本模式（30 分钟）
2. **再看 ecommerce-rag-agent**：掌握 LangGraph 工作流和状态管理（1 小时）
3. **最后看 retailGPT**：学习购物车完整实现和 Function Calling（1 小时）

学习完成后，建议以 ecommerce-rag-agent 为基础框架，融入 retailGPT 的购物车逻辑，自建完整的 Python 后端服务。
